Known Limitations

What is still being worked on.

UIWeaver is experimental. These are the known areas that need more work before UIWeaver is ready for complex production UIs.

Slot layout synchronization

Minecraft requires Slot objects to be registered at exact pixel positions matching what the screen renders. UIWeaver computes slot positions from the layout pass, but any inconsistency between the layout result and the registered slot positions causes misaligned click targets and item rendering issues.

This is the highest priority area for improvement.

Tabs

Inactive tabs should not participate in layout, receive input, or be considered by the inspector unless explicitly debugging. The current implementation handles this in most cases but can produce stale child bounds after switching tabs quickly.

Styling

There is no styling API yet. Component appearance is determined by the default renderer. Planned:

  • Panel backgrounds and borders
  • Hover and active states
  • Spacing tokens
  • Theme variants
  • Reusable style presets

Scroll and large inventories

The current scrollPanel works for general content but does not support virtual scrolling over very large item handlers (thousands of slots). Implementing that requires a custom slot remapping layer outside the standard UIWeaver component model.

API stability

UIWeaver is at v0.1.0. Method names and package structure may change in future versions. Check the GitHub changelog before upgrading.
